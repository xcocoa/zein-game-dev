# StoryForge — AI 互动叙事引擎

> 把小说变成可玩的视觉小说游戏，让每个有故事的人都能零成本做出自己的游戏。

---

## 这是什么

StoryForge 是一个 AI 驱动的内容转换工具：**输入一篇中文小说，自动输出一个可玩的视觉小说（Galgame）游戏**。

整个过程全自动——AI 负责提取角色、识别场景、分析对话、生成互动选择和多结局分支，最终输出完整的游戏项目。

**30 秒了解效果**：我们用芥川龙之介的《罗生门》（3000 字）做了测试，从导入到生成可玩游戏，全程自动完成。游戏包含 2 个角色、3 个章节、2 个互动选择点、4 个不同结局。

---

## 技术亮点

这不是简单的"套个 LLM API"，而是一套完整的内容理解与代码生成管线：

### 6 阶段 LLM 编排管线

```
小说文本
  → Step 1: 角色提取（识别人物、性格、关系）
  → Step 2: 场景识别（地点、氛围、时间）
  → Step 3: 结构分析（对话/叙述/动作分类）
  → Step 4: 选择生成（互动决策点、变量效果）
  → Step 5: 结局设计（多结局、触发条件）
  → Step 6: 章节划分（节奏把控）
  → 结构化故事数据（JSON）
```

### 编译器级架构

借鉴编译器设计思想，采用三层架构：

```
Parser（解析器）→ AST（抽象语法树）→ Code Generator（代码生成器）
```

- **Parser 层**：LLM 将自然语言转为结构化数据（Pydantic 模型）
- **AST 层**：18 种节点类型表示游戏逻辑（对话、选择、条件、跳转……）
- **Generator 层**：AST → 目标代码（当前支持 Ren'Py，架构可扩展）

这个中间表示层让我们可以轻松扩展到不同的输出目标（Web 播放器、Unity、Godot 等），而不用重写解析逻辑。

### 当前技术栈

| 层级 | 技术 |
|------|------|
| LLM 解析 | Ollama（本地）/ 云端 API（计划中） |
| 数据模型 | Python + Pydantic |
| AST 节点 | Python dataclass，18 种 Ren'Py 语句类型 |
| 代码生成 | 自研 Ren'Py 生成器 |
| 项目构建 | 自动生成完整可运行的 Ren'Py 项目（含中文字体、GUI 屏幕） |
| 测试覆盖 | 104 个单元测试 |
| CLI | Typer + Rich |

## 接下来要做什么

我们的目标是把 StoryForge 从命令行工具变成一个 **Web 平台**——用户在浏览器里上传小说，等待几分钟，就能在线玩生成的游戏，还能分享给朋友。

### 核心任务

**Web 化**
- 后端 API 服务（FastAPI，包装现有管线）
- 异步任务队列（处理生成任务）
- Web 视觉小说播放器（在浏览器里直接玩，不需要装 Ren'Py）

**AI 增强**
- 接入云端 LLM API（DeepSeek / Qwen），不再依赖本地模型
- AI 生成角色立绘和场景背景（接入 SD/FLUX）
- 优化 prompt，提升生成质量的稳定性

**产品化**
- 可视化剧情编辑器（修改 AI 生成的剧情线）
- 作品广场（发布、浏览、分享）
- 移动端适配

## 寻找队友

我们正在组建一个 3-5 人的小团队，寻找以下方向的伙伴：

### AI 工程师

负责 LLM 管线优化、云端模型接入、AI 生图集成。

如果你：
- 有 LLM 应用开发经验（不限框架）
- 对 prompt engineering 有实践心得
- 想做一个"效果看得见、玩得到"的 AI 项目，而不是又一个 chatbot

这里有真实的技术挑战：从非结构化小说中抽取角色关系图、情节分支树、情感变量系统——比做 RAG 有意思多了。

### 前端工程师

负责 Web 视觉小说播放器、剧情可视化编辑器、用户页面。

如果你：
- React/Vue 熟练，有独立做过完整项目
- 对游戏或互动体验感兴趣
- 想做有技术深度的前端，而不是普通 CRUD

你将负责：对话框动画系统、分支选择交互、变量追踪引擎、节点图剧情编辑器。这些都是游戏级的前端工程。

### 后端工程师

负责 API 服务、任务队列、用户系统、部署运维。

如果你：
- 熟悉 Python（FastAPI）或 Go
- 有异步任务处理经验
- 想从 0 到 1 搭建一个 AI SaaS 架构

你不是在给一个现有系统加功能，而是设计一个全新产品的后端。
